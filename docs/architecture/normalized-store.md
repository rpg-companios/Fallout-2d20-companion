# Normalized Character Store — Architecture

## Overview

Character state is managed by a Zustand store (`src/store/characterStore.js`) with localStorage persistence. All mutable runtime data lives in this store; `CharacterContext` is a legacy wrapper that still provides UI state (character name, origin, trait, etc.) not yet migrated.

## State Shape

```
{
  attributes : { [attrId: string]: Parameter<number> }
  skills     : { [skillId: string]: Parameter<number> }
  items      : { [instanceId: string]: NormalizedItem }
  effects    : { [effectId: string]: NormalizedEffect }
}
```

### Parameter<T>

```js
{
  base      : T,           // unmodified base value
  modifiers : Modifier[],  // list of active modifiers
  total     : T,           // base + sum(modifiers)
}
```

### NormalizedItem

```js
{
  id        : string,   // instance key (UUID-like, generated)
  weaponId  : string,   // canonical catalog ID (see "Item identity" below)
  name      : string,
  itemType  : string,   // 'weapon' | 'armor' | 'clothing' | 'chem' | 'ammo' | 'misc' | ...
  equipped  : boolean,
  quantity  : number,
  stackKey  : string,   // same for identical items (canonicalId + mods)
  appliedMods : { [slot]: modId },
  // combat parameters (optional, normalized):
  damage, fireRate, range, damageType,
  physicalDamageRating, energyDamageRating, radiationDamageRating,
  // consumable effects (optional):
  positiveEffect, negativeEffect, addictionLevel, ...
}
```

## Item Identity

Every item entering the store must have a *canonical ID*. `addNewItem` resolves it from
the first truthy field (in priority order):

```
item.weaponId → item.id → item.code → item.itemId → item.armorId → item.clothingId
```

The canonical ID is stored as `normalizedItem.weaponId` so WeaponsAndArmorScreen and
other consumers can look up catalog data using the same ID.

The *instance key* (dict key in `items`) is generated by:
- No mods: `canonicalId` (e.g. `weapon_10mm_pistol`)
- With mods: `canonicalId_mods_mod1_mod2` (e.g. `weapon_10mm_pistol_mods_mod_004_mod_017`)

This allows the same weapon with different mod configurations to coexist as separate
inventory entries.

## Kit → Inventory Data Flow

```
Kit JSON (data/equipmentKits/*.json)
  └─► resolveKitItems()          [domain/kitResolver.js]
        Async: catalog lookup for each item
        armor/clothing/misc → armorIndex / clothesMap / consumables
        weapons → weapon catalog + mod resolution
        ammo    → ammo catalog (also adds id: ammoId)
  └─► flattenKitItems()          [EquipmentKitModal.js]
        Flattens choice entries using user-selected options
  └─► toInventoryItems()         [EquipmentKitModal.js]
        Weapons  → { weaponId, id, name, appliedMods, ... }
        Non-weapons → spread resolved item (has catalog id)
        Ammo     → { id: ammoId, name, quantity, itemType: 'ammo', ... }
  └─► summarizeItems()           [EquipmentKitModal.js]
        Strips currency items, calculates total weight/price/caps
  └─► CharacterScreen.handleSelectKit()
        Calls addNewItem(item) for each finalItem
  └─► addNewItem()               [characterStore.js]
        Resolves canonical ID, normalizes parameters, stores in items dict
  └─► InventoryScreen             reads via selectItemsByEquipped(state, false)
```

## Persistence

- **Key**: `character-store` (localStorage)
- **Middleware**: Zustand `persist` (partial state: attributes, skills, items, effects)
- **On load**: `normalizeForStore(data)` converts legacy array-based saved data to
  normalized dict format
- **On save**: `denormalizeForSave(storeState)` converts back to legacy format for DB
  compatibility (`CharacterContext.buildSnapshot` → `mergeSnapshotWithStoreData`)

## Known Limitations / Future Work

- `CharacterContext` still owns UI state (name, origin, trait, equipment metadata,
  effects array, caps, health) and deprecated setters (`setAttributes`,
  `setEquippedWeapons`). Full migration would eliminate CharacterContext entirely.
- `setEquippedWeapons` / `setAttributes` in CharacterContext are still called by
  several screens — they shadow the Zustand store for those paths.
- Items with catalog IDs that don't exist in the equipment catalog (e.g. a kit JSON
  referencing a non-existent armor ID) are stored using their raw identifier as the
  canonical ID, so they appear in the inventory with their original id but without
  resolved catalog stats.
