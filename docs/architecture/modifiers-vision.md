# Видение: единая система модификаторов (черновик, НЕ реализовано)

## Принцип (от владельца)
«Любое окно с данными в программе подвержено влиянию модификаторов, даже сами
модификаторы подвержены другим модификаторам.»

То есть: **всё значение в игре = базовое значение + стек модификаторов → итог.**
Источников модификаторов много, они складываются, и могут влиять друг на друга.

## Что УЖЕ есть (зародыш)
В Zustand-сторе атрибуты/навыки/предметы уже хранятся как
`Parameter = { base, modifiers: [...], total }` (см. src/store/resolvers.js,
characterStore.js). Это фактически фундамент будущей системы — но:
- применяется не ко всему (производные, химия, черты вливаются разными путями);
- источники (черты, перки, эффекты, экипировка) НЕ приводятся к единому виду модификатора.

## Целевая идея (предварительно)
Единый тип модификатора, самоописательный:
```jsonc
{
  source: "trait:supermutant",      // кто наложил (для снятия/трассировки)
  target: "attribute.STR",          // на что влияет (путь к значению)
  kind:   "add" | "set" | "min" | "max" | "mult" | "flag" | "immunity" | ...,
  value:  2,
  priority: 0,                      // порядок применения (модификаторы на модификаторы)
  condition?: {...}                 // опционально: когда активен
}
```
Источники (черты, перки, эффекты, экипировка, химия) выдают **массив таких модификаторов**.
Движок собирает все активные → группирует по target → применяет по kind/priority → total.

## Почему это большой проект (честно)
- Затрагивает ВСЁ: атрибуты, навыки, производные (вес/HP/защита/инициатива/удача),
  параметры предметов, иммунитеты, лимиты, кол-во тегов/перков.
- Требует: единый реестр target-путей, порядок применения (add vs set vs mult vs min/max),
  разрешение «модификатор на модификатор», трассировку источников (чтобы снимать при смене черты).
- Миграция данных трейтов/перков/эффектов в новый вид.
- Это отдельный крупный этап — проектировать тщательно, не наскоком.

## Текущее решение по трейтам (временно, осознанно)
Формат `attributes.STR = { baseBonus, min, max }` оставлен как есть (фикс #6 его поддерживает).
Это смешение «эффект + ограничение» в одном поле — в будущей системе разложится на
два модификатора (`kind:add` и `kind:min/max`). Пока не трогаем.

## Когда возьмёмся — порядок
1. Спроектировать схему модификатора + список target-путей (согласовать с владельцем).
2. Расширить Parameter-модель/резолверы на все значения.
3. Привести источники (черты→modifiers[], перки→, эффекты→, экипировка→) к единому виду.
4. Миграция данных + тесты + трассировка источников.
