# Схема: Modifiers (модификаторы) — универсальный каталог

> Этот документ — **каталог всех известных модификаторов**: что каждый делает, где живёт,
> как применяется. Цель — любой ИИ или человек может заглянуть сюда, понять существующие
> модификаторы, и спроектировать новый (или повторно использовать готовый) по тому же шаблону.

## 0. Что такое модификатор

Модификатор — это декларативное поле в данных, описывающее правило. Движок (в `domain/`)
читает его и применяет к персонажу. Все модификаторы:

- **camelCase** в ключах (правило проекта).
- Лежат в `modifiers: { ... }` объекта источника.
- Имеют **одну** точку истины: либо поле на трейте, либо на ориджине, либо на перке,
  и т.д. Дубликаты между источниками — недопустимы (см. T-1 в `02-traits.md`).
- Имеют **описанный приоритет** при коллизии (что перебивает что).

Источники модификаторов (все живут в `modifiers: { ... }`):

| Источник | Где живёт | Когда применяется |
|---|---|---|
| `Origin` | `data/origins/origins.json` | на персонаже всегда (если origin выбран) |
| `Trait` | `data/traits/traits.json` | после выбора трейты (только выбранной) |
| `Perk` | (будущее) | после выбора перка |
| `Consumable effect` | (будущее) | во время действия эффекта (на сцене) |
| `Item` (броня/оружие) | `data/equipment/{armor,weapons,robot}.json` | пока предмет экипирован |

Все модификаторы выше **одинаковы по форме** (`{key: value}`), но **различаются по
приоритету** при слиянии (см. раздел «Приоритеты» в конце).

## 1. Каталог модификаторов

Формат записи: **key** — тип значения — scope — источник — поведение.

### 1.1. Атрибуты

#### `attributeBonus`
- **Тип:** `{ [ATTR]: number }` где `ATTR ∈ {STR,END,PER,AGI,INT,CHA,LCK}`, `number` может быть отрицательным.
- **Где:** trait, perk.
- **Поведение:** +N к базовому значению атрибута. Бонус и лимит независимы — трейт
  может дать бонус без лимита.
- **Пример:**
  ```jsonc
  "attributeBonus": { "STR": 2, "END": 2 }
  ```

#### `attributeLimits`
- **Тип:** `{ [ATTR]: { min?: number, max?: number } }`.
- **Где:** trait, perk.
- **Поведение:** расширяет нижнюю/верхнюю границу атрибута. min/max могут быть
  заданы независимо (`{ max: 6 }` — только верхняя граница).
- **Пример:**
  ```jsonc
  "attributeLimits": { "STR": { "min": 6, "max": 12 }, "CHA": { "max": 6 } }
  ```

#### `attributePointsBonus`
- **Тип:** `number`.
- **Где:** trait, perk.
- **Поведение:** +N очков распределения сверх базовых 12.

### 1.2. Навыки

#### `forcedSkills`
- **Тип:** `string[]` (имена навыков — локализованные сейчас, TODO T-3 → canonical).
- **Где:** trait.
- **Поведение:** навыки из списка становятся тегами (стартовое значение 2).

#### `extraSkills`
- **Тип:** `number`.
- **Где:** trait.
- **Поведение:** +N дополнительных тегов на выбор игрока.

#### `skillModifiers`
- **Тип:** `{ [skillName]: number }` (TODO T-3 → canonical keys).
- **Где:** trait.
- **Поведение:** +N к значению навыка.

#### `skillMaxValue`
- **Тип:** `number` (по умолчанию 6).
- **Где:** trait.
- **Поведение:** верхняя граница навыка; на 1 уровне ≤ 3.

### 1.3. Перки

#### `extraPerks` / `extraPerkSlots`
- **Тип:** `number`.
- **Где:** trait.
- **Поведение:** +N перк-слотов на выбор.

### 1.4. Carry weight (переносимый вес)

#### `carryWeightLimit`
- **Тип:** `number`.
- **Где:** **trait (робот)** И **body catalog** (`data/equipment/robot/robotbody.json`).
- **Поведение:** базовый лимит переносимого веса **для роботов**. Один ключ в обоих
  источниках — body конкретизирует/перебивает trait:
  1. Если в слоте тела (`body`/`chassis`/`thruster`) есть `limb.carryWeightLimit` → это.
  2. Иначе `trait.modifiers.carryWeightLimit` (по умолчанию 150).
- **Пример (trait):**
  ```jsonc
  "carryWeightLimit": 225  // протекрон в базе несёт 225
  ```
- **Пример (body):**
  ```jsonc
  "carryWeightLimit": 200  // конкретный корпус несёт 200
  ```

#### `carryWeightStrengthMultiplier`
- **Тип:** `number` (по умолчанию 10).
- **Где:** trait (human).
- **Поведение:** множитель STR в формуле веса для людей:
  `weight = base + STR × multiplier + armor_mods`.

#### `carryWeight` *(legacy, additive)*
- **Тип:** `number`.
- **Где:** trait (human, опц., сейчас не используется ни одним трейтом).
- **Поведение:** +N к базовому лимиту **поверх** формулы (отдельный аддитивный модификатор).
  **Семантически отличается от `carryWeightLimit`** (которое = база для роботов).
- **TODO:** переименовать в `carryWeightBonus` для ясности.

#### `carryWeightModifier` *(на предметах)*
- **Тип:** `number` (может быть отрицательным).
- **Где:** armor, clothing, plating/frame/robot armor (catalog).
- **Поведение:** +N к итоговому carry weight **для конкретного слота** (надевается).

### 1.5. Производные / ресурсы

#### `luckMaxDelta`
- **Тип:** `number` (может быть отрицательным).
- **Где:** trait.
- **Поведение:** сдвиг максимума очков удачи.

#### `meleeBonusDelta`
- **Тип:** `number` (может быть отрицательным).
- **Где:** trait.
- **Поведение:** +N к бонусу ближнего боя (сверх STR-ступеней).

### 1.6. Иммунитеты / сопротивления / эффекты

#### `immunities`
- **Тип:** `string[]` (значения: `'disease'`, `'radiation'`, `'poison'`, ...).
- **Где:** **trait**, **origin** (`origin.immunities` — опц.).
- **Поведение:** явный список иммунитетов. Если есть — добавляются к проверке
  иммунитетов персонажа. **Не type-derived** (см. отклонение в `01-origins.md`).
- **Пример:**
  ```jsonc
  "immunities": ["radiation", "poison"]
  ```

#### `radiationResistance`
- **Тип:** `number`.
- **Где:** trait.
- **Поведение:** сопротивление радиации (не иммунитет, а снижение урона).

#### `effects`
- **Тип:** `string[]` (именованные флаги).
- **Где:** trait.
- **Поведение:** именованный эффект-флаг. Семантика зависит от самого флага.
- **TODO (отдельная задача):** аудит 28+ эффектов (`slow_aging` уже дропнут как мусор).

### 1.7. Универсальные модификаторы (трейт / перк / химия)

Эти модификаторы **применимы в любом источнике** (трейт, перк, эффект химии) и
движок суммирует их из всех активных источников.

#### `weaponDamageBonus`
- **Тип:** `Array<{ weaponIds?: string[], weaponId?: string, skillKey?: string, bonus: number }>`.
- **Где:** trait, perk, consumable effect (в будущем).
- **Поведение:** запись матчит оружие по `weaponIds[]`, `weaponId` или `skillKey`
  (UPPER_SNAKE_CASE из `ALL_SKILL_KEYS`). При совпадении прибавляет `bonus` (может
  быть отрицательным — штраф). Несколько записей в массиве суммируются для одного
  оружия. Движок суммирует из всех активных источников через `getWeaponDamageBonusFromSources`.
- **Пример:**
  ```jsonc
  "weaponDamageBonus": [
    { "weaponIds": ["weapon_10mm_pistol"], "bonus": 1 },
    { "skillKey": "BIG_GUNS", "bonus": -1 }
  ]
  ```

### 1.8. Структурные (метаданные трейта)

#### `isMultiTrait`
- **Тип:** `boolean`.
- **Где:** trait.
- **Поведение:** маркер «это контейнер с под-чертами» (NCR, Survivor, Savage).

#### `subTraitIds`
- **Тип:** `string[]`.
- **Где:** trait (multi-trait).
- **Поведение:** список id под-черт для выбора.

#### `goodSoulSkills`
- **Тип:** `string[]` (canonical skill keys).
- **Где:** trait (`ncr-good-soul`).
- **Поведение:** список навыков, которые Good Soul даёт на выбор. Источник истины
  для `goodSoulGroup` (раньше хардкодился в `SurvivorModal.js`).

## 2. Приоритеты при коллизии

Если один и тот же атрибут модифицируется несколькими источниками одновременно:

| Сценарий | Приоритет |
|---|---|
| **Атрибуты** (attributeBonus, attributeLimits) | additive — все источники складываются (если не противоречат) |
| **Иммунитеты** | additive (union множеств) — повторы не дублируются |
| **Иммунитеты по типу** | ❌ НЕ применяется (отклонение от `01-origins.md`); только явные списки |
| **Carry weight limit (робот)** | body.carryWeightLimit → trait.carryWeightLimit → 150 |
| **Carry weight limit (human)** | база = 150 + STR × multiplier (multiplier из trait) + additive `carryWeight` |
| **Carry weight modifier (предмет)** | additive per slot (броня + одежда для людей; plating/armor/frame для робо-слотов) |
| **weaponDamageBonus** | additive — все активные источники суммируются |
| **skillMaxValue** | берётся максимум (расширяет потолок, не сужает) |
| **forcedSkills / subTraitIds** | union |

## 3. Шаблон: как добавить новый модификатор

Если нужно ввести новый модификатор (например, `criticalHitBonus`):

1. **Опиши в этом документе** (раздел 1.x):
   - Имя ключа (camelCase).
   - Тип значения.
   - Где живёт (trait / perk / origin / item / chem effect).
   - Поведение (как движок применяет).
   - Приоритет при коллизии (раздел 2).

2. **Добавь в `data/...`** — фикстуры, использующие модификатор.

3. **Добавь хелпер в `domain/`** — чистую функцию (например, `getCriticalHitBonus(character)`).

4. **Добавь тест в `domain/*.test.js`** — поведение функции, edge cases.

5. **Используй в UI / резолверах** через хелпер.

6. **Если нужен UI-текст** — вынеси в i18n (`i18n/{locale}/...`), не хардкодь.

## 4. Связь с будущей системой модификаторов

Этот каталог — **текущее состояние** декларативных правил. Долгосрочная цель
(см. `docs/architecture/modifiers-vision.md`) — единая система модификаторов
с `source` / `target` / `kind` / `priority` и полным трассированием.

До тех пор этот документ — **контракт** между data и движком. Любое поле в
`modifiers: { ... }` любого источника должно быть описано здесь.
