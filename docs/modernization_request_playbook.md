# Modernization Request Playbook

Этот документ фиксирует правила, которые нужно применять при любом новом запросе на модернизацию кода.

## Цель

Сделать изменения предсказуемыми и единообразными:
- переход на `camelCase`;
- удаление обратной совместимости (если не запрошено иное);
- подготовка данных для i18n;
- минимизация лишних парсеров/резолверов;
- поддержка новой структуры каталогов (экраны/эффекты/переводы/модалки).

## Базовые принципы

1. **Новые поля/ключи только в `camelCase`**.
2. **Не добавлять legacy-алиасы**, если это не отдельная задача.
3. **Все пользовательские строки выносить в i18n** (никаких хардкодов в компонентах).
4. **Не плодить новые normalizer/parser/resolver слои**, если можно использовать уже существующие данные напрямую.
5. **Изменения по экрану делать комплектно**: UI + логика + переводы + модалки + импорты/пути.

## Целевая структура (ориентир)

При рефакторинге экранов придерживаться такой схемы:

- `components/screens/<ScreenName>/`
  - `<ScreenName>.js` (или `index.js`)
  - `logic/`
  - `modals/`
  - `effects/` (если есть эффекты, специфичные для экрана)

- `i18n/<locale>/screens/<screenName>/`
  - `screen.json` — тексты самого экрана
  - `modals/<ModalName>.json` — тексты модалок экрана

- `i18n/<locale>/effects/`
  - файлы переводов, относящиеся к эффектам

> Если в текущем коде еще остались старые плоские файлы (`i18n/<locale>/InventoryScreen.json` и т.д.),
> при модернизации экрана переносить их в новую структуру и обновлять импорты.

## Алгоритм при новом запросе

### 1) Определить границы задачи
- Какой экран/модуль меняется.
- Какие сущности затрагиваются (атрибуты, навыки, экстра-навыки, оружие, броня, препараты, происхождения, черты).

### 2) Обновить файловую структуру
- Создать недостающие каталоги для экрана:
  - `components/screens/<ScreenName>/logic`
  - `components/screens/<ScreenName>/modals`
  - `components/screens/<ScreenName>/effects` (по необходимости)
- Создать каталоги переводов:
  - `i18n/ru-RU/screens/<screenName>/`
  - `i18n/en-EN/screens/<screenName>/`
  - `i18n/*/screens/<screenName>/modals/` (если есть модалки)

### 3) Перенести и нормализовать ключи i18n
- Перенести строки экрана в `screen.json`.
- Перенести строки каждой модалки в отдельный json в `modals/`.
- Привести ключи к `camelCase`.
- В обоих языках держать одинаковую структуру и набор ключей.

### 4) Обновить импорты и пути
- Обновить импорты переводов в коде экрана и модалок.
- Проверить пути в агрегаторах i18n (например, `i18n/locale.js`, `i18n/appI18n.js`, или актуальный entrypoint).
- Удалить/отвязать старые пути, если перевод уже переехал.

### 5) Упростить поток данных
- Удалить неиспользуемые адаптеры legacy-полей.
- Не добавлять временные мапперы форматов без явной необходимости.
- Хранить и передавать данные в одном согласованном формате (`camelCase`).

### 6) Подготовить расширение механик
Если задача касается механик (модификаторы, бонусы/штрафы, происхождения, черты):
- выделять логику в отдельные функции/модули внутри экрана или доменной папки;
- описывать модификаторы декларативно (данные + обработчик), чтобы было проще расширять;
- избегать зашивания правил прямо в JSX/рендер.

## Чеклист для запроса вида «перевести экран Inventory»

1. Создать:
   - `i18n/ru-RU/screens/inventory/screen.json`
   - `i18n/en-EN/screens/inventory/screen.json`
2. Для каждой модалки Inventory создать отдельные файлы переводов:
   - `i18n/ru-RU/screens/inventory/modals/<ModalName>.json`
   - `i18n/en-EN/screens/inventory/modals/<ModalName>.json`
3. Перенести строки из старых файлов/хардкодов в новые json.
4. Обновить вызовы переводов в:
   - `components/screens/InventoryScreen/*`
   - `components/screens/InventoryScreen/modals/*`
5. Обновить регистрационные пути i18n в центральной конфигурации.
6. Удалить устаревшие плоские json экрана (если больше не используются).
7. Проверить, что ключи едины и в `camelCase`.

## Что считать завершенной модернизацией

Задача считается завершенной, когда:
- структура файлов экрана и переводов приведена к целевой;
- старые пути и legacy-форматы не используются;
- все новые/измененные ключи имеют `camelCase`;
- тексты экрана/модалок полностью вынесены в i18n;
- код не получил лишних промежуточных parser/resolver слоев.
