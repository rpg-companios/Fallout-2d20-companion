## Дизайн-задание: Переработка экипировки роботов, динамическое оружие и стартовые комплекты

### 1. Цели

1. **Создать для роботов самостоятельную систему слотов**, параллельную человеческой броне, но без ограничений людского интерфейса.
2. **Автоматически заполнять слоты робота** при его создании/смене шасси из **комплекта снаряжения** (`equipmentKit`), включающего конечности, оружие, защиту и модули. Никакой ручной экипировки конечностей из инвентаря – всё сразу готово.
3. Поддерживать **три слоя защиты** на каждом слоте (броня, обшивка, рама) с правилами совместимости.
4. **Полностью убрать ограничение на количество оружия** (динамический массив `equippedWeapons`), сохранив обратную совместимость для людей.
5. **Манипуляторы роботов остаются видимым оружием** даже при установке дополнительного оружия (динамический список позволяет иметь оба).
6. **Люди автоматически получают «Рукопашную атаку»** как неубираемое встроенное оружие.

---

### 2. Новая структура данных (CharacterContext)

#### 2.1. `equippedWeapons`
- **Тип:** массив объектов оружия (без `null`).
- **У людей:** минимум один элемент – встроенная рукопашная атака (`id: "unarmed_human"`), не подлежащая удалению.
- **У роботов:** собирается динамически из всех активных источников (встроенное, манипуляторное, удерживаемое).

#### 2.2. `equippedRobotSlots` (только для роботов)
Объект с фиксированными ключами `head`, `body`, `leftArm`, `rightArm`, `leftLeg`, `rightLeg`.  
Каждое значение – объект:
```ts
{
  limb: object | null,     // конечность (robotHead, robotArm, robotLeg, robotBody)
  armor: object | null,    // броня
  plating: object | null,  // обшивка
  frame: object | null,    // рама
  heldWeapon: object | null // оружие, удерживаемое манипулятором (только для рук)
}
```

#### 2.3. Определение «робот»
Единая функция `isRobotCharacter(character)` проверяет:
- `origin.isRobot`
- `trait.modifiers.isRobot`
- `trait.modifiers.robotBodyPlan` (не пусто)

#### 2.4. `equippedArmor`
Без изменений, используется только людьми. Роботы его игнорируют.

### 3. Стартовые комплекты (`equipmentKit`)

#### 3.1. Источники слотов
При создании робота **все** слоты конечностей берутся из комплекта снаряжения для данного `robotBodyPlan` и варианта (например, `protectron_standard`).

В комплекте **явно перечисляются** конечности, оружие, защита и модули. Пример структуры уже передан тобой. Дополнительные поля предметов:
- **Для конечности:** `itemType: "robotArm" | "robotHead" | "robotBody" | "robotLeg"`, `id`, `slots` (массив допустимых ключей, например `["leftArm", "rightArm"]`), `builtinWeaponId` (опционально), `canHoldWeapons`, `maxHandelWeaponWeight`, `excludeTwoHanded`.
- **Для оружия:** помимо стандартных полей, может содержать:
  - `requiresWeaponId` – указывает, что оружие должно быть помещено в `heldWeapon` слота, где конечность имеет такой же `id` (т.е. требует конкретный манипулятор).
  - `slot` – явное указание слота (`"left"`, `"right"`).
  - `replacesArm: true` – оружие само становится конечностью (занимает слот `limb`, предыдущая конечность заменяется). В этом случае `itemType` может быть `weapon`, но обработчик поймёт, что это замена.

#### 3.2. Алгоритм инициализации (`initRobotSlots`)

1. Создаётся пустой объект `slots` с ключами `head`, `body`, `leftArm`, `rightArm`, `leftLeg`, `rightLeg`, все поля `null`.
2. Обрабатываются предметы из комплекта в порядке их перечисления (с учётом `choice`, `rollTable` – логика раскрытия делается до заполнения слотов).
3. Для каждого предмета:
   - Если `itemType` – конечность (`robotArm`, `robotHead`, `robotBody`, `robotLeg`):
     - Определяется целевой слот (если `slots` указаны, берётся первый подходящий свободный; можно учесть `slot` для рук).
     - Если слот занят, может быть замена (по правилам совместимости) или ошибка.
     - В `slots[slotKey].limb` записывается объект конечности.
   - Если `itemType` – оружие с `replacesArm: true`:
     - Аналогично конечности, занимает слот `limb` в выбранной руке (предыдущая конечность удаляется).
     - Встроенное оружие этой «конечности» добавляется в массив оружия.
   - Если `itemType` – оружие без `replacesArm`:
     - Если указан `requiresWeaponId`, ищется слот с конечностью-манипулятором, у которой `id === requiresWeaponId`, и `canHoldWeapons === true`.
     - Если указан `slot`, используется конкретный слот (проверяется, что там манипулятор).
     - Если нет ни того, ни другого – ошибка или трактуется как встроенное оружие для слота? Уточнить: в твоих примерах такого нет.
     - Найденный слот: `slots[slotKey].heldWeapon = weaponObject`.
   - Если `itemType` – защита (`plating`, `armor`, `frame`):
     - Устанавливается в соответствующий слой всех слотов, указанных в `protectedAreas` (или `slot`). В примерах – обычно `body`.
     - Проверка конфликтов слоёв (если plating несовместим с frame – ошибка).
   - Если `itemType` – модуль (`module`):
     - Добавляется в отдельный массив `equippedRobotModules` (будет создан).
   - Прочие предметы (химия, еда, ремнаборы, крышки) – в инвентарь (массив `equipment.items`).
4. После заполнения всех слотов вычисляется массив `equippedWeapons`:
   - Для каждой конечности с `builtinWeaponId` (или встроенным оружием) – добавляется объект оружия.
   - Для каждого `heldWeapon` – добавляется объект оружия.
   - Для каждого манипулятора без `heldWeapon` (имеющего `builtinManipulator: true`) – добавляется манипулятор как оружие.
   - Все добавленные оружия помечаются идентификатором слота (для UI – подпись «из правой руки»).

Этот алгоритм гарантирует, что робот после создания сразу имеет полный набор слотов и оружия, а игроку не нужно ничего надевать.

### 4. Динамическое оружие (общее)

- `equippedWeapons` – массив, может содержать любое количество элементов.
- **Добавление:** в инвентаре нажатием «Экипировать». Для роботов – выбор подходящего манипулятора (диалог, как сейчас выбор руки для наручей). Для людей – просто push в массив.
- **Удаление:** кнопка «Снять» в инвентаре; для роботов – очищает `heldWeapon` слота (если оружие встроенное/манипуляторное – кнопка неактивна).
- **Рукопашная атака людей:** присутствует всегда, не удаляется.

### 5. Изменения UI

#### 5.1. `WeaponsAndArmorScreen`
- **Для людей:** старый рендеринг ArmorPart + массив WeaponCard.
- **Для роботов:**
  - В блоке брони рендерятся `RobotSlot` для каждого непустого ключа `head`, `body`, `leftArm`, `rightArm`, `leftLeg`, `rightLeg`.
  - Каждый `RobotSlot` показывает:
    - Название слота (можно с учётом нестандартного шасси: «Рука 1», «Рука 2», «Двигатель»).
    - Название конечности.
    - Индикаторы слоёв защиты (броня, обшивка, рама) со значениями.
    - Кнопка «Модернизировать» → открывает модальное окно замены конечности (список совместимых предметов данного типа из каталога).
    - Если слот даёт оружие (встроенное/удерживаемое) – показывается краткая информация об уроне, клик ведёт на карточку оружия.
  - Блок оружия: динамический список всех `WeaponCard` с указанием слота-источника (например, «R Arm»). Карточки оружия работают как раньше, включая модификации.

#### 5.2. `InventoryScreen`
- **Для людей:** всё по-старому, но с возможностью экипировать много оружия.
- **Для роботов:**
  - Предметы-конечности (`robotArm`, `robotHead`, …) в инвентаре видны только если станадартную конченость из комплекта снаряжения заменили на экране оружия и брони.
  - Обычное оружие (не роботизированное) можно экипировать, если есть манипулятор: кнопка «Экипировать» вызывает диалог выбора слота.
  - Расходники – как обычно.

### 6. Доменный слой (новые чистые функции)

Модуль `domain/robotEquip.js`:

- `isRobotCharacter(character)`
- `initRobotSlots(bodyPlan, kitId, catalog, randomSeed)` → `{ slots, weapons, inventoryItems, modules }`
- `getBuiltinWeaponsFromSlots(slots)` → массив оружия
- `canReplaceLimb(slot, newLimb, character)` → boolean
- `applyLimbReplacement(slots, slotKey, newLimb)` → новые слоты + обновлённые оружия
- `canEquipWeaponToSlot(weapon, slot, character)` → boolean
- `canEquipRobotArmor(armorItem, slot, character)` → boolean

### 7. Доработка модалки получения предмета в инвентаре

В модалке "получить добычу" при получении предмета должно отображаться возможное количество для получения. Не 1 патрон за раз, а пользователь сам пишет, сколько за раз. Не 1 пистолет, а пользователь сам указывает количество. И именно такое количество попадает в инвентарь.

1. **Рефакторинг `equippedWeapons`** – замена на динамический массив, добавление встроенного оружия людей.

3. **Добавление `equippedRobotSlots` и `equippedRobotModules` в контекст**, интеграция в создание персонажа.
