# Требования: Нормализованное хранение всех игровых параметров через Zustand Store

## Введение

Проект — мобильное RPG-приложение-компаньон для настольной игры Fallout. Текущая архитектура хранения игровых параметров разрозненна и приводит к рассинхронизации:

### Проблемы текущей модели:

1. **Предметы** (`equipment`, `equippedWeapons`, `equippedArmor`):
   - При экипировке предмет копируется из инвентаря в экипировку (создается новая копия с `uniqueId`)
   - Модификации хранятся отдельно в `modifiedItems` (Map), но не интегрированы с базовыми данными
   - Изменения в одном экране не отражаются в других без ручной синхронизации

2. **Атрибуты и навыки** (`attributes`, `skills`):
   - Базовые значения хранятся в `CharacterContext`, но не учитывается влияние эффектов
   - Перки могут увеличивать атрибуты (`commitAttributeChanges`), но это не интегрировано с `timedEffects`
   - Эффекты (`activeTimedEffects`) применяются отдельно и не пересчитывают производные параметры ( здоровье, защита, инициатива )

3. **Эффекты** (`timedEffects`, `activeTimedEffects`):
   - Истечение эффектов обрабатывается в `pruneExpiredTimedEffects`, но не вызывает пересчет атрибутов/параметров
   - Бонусы от эффектов (`timedMaxHpBonus`, `timedDR`) вычисляются локально в каждом экране, а не интегрированы в итоговые значения

4. **Перки и черты** (`selectedPerks`, `trait`):
   - Модификаторы применяются в `CharacterScreen` при изменении черты, но не отслеживаются в едином месте
   - Требования перков проверяются локально (`meetsPerkRequirements`), но базовые атрибуты не обновляются динамически

**Корневая причина**: Нет единой модели, где каждый параметр (сила, ловкость, урон оружия, скорость стрельбы, защита) хранится в одном экземпляре, а все эффекты/модификаторы просто добавляют/вычитают значения. Каждый экран работает со своей копией данных.

---

### Цель рефакторинга:

Создать нормализованную модель данных через Zustand Store, где:
1. Каждый параметр (атрибут, навык, поле предмета, параметр эффекта) существует в одном экземпляре
2. Все эффекты (черты, перки, timed-эффекты, моды оружия) применяются как модификаторы к базовым значениям
3. При изменении любого элемента (эффект истек, перк применен, предмет экипирован) итоговые значения пересчитываются автоматически
4. Экраны просто отображают итоговые значения из глобального хранилища — они всегда актуальны

---

## Требования

### Требование 1: Единый источник истины для всех параметров

**User Story:** Как разработчик, я хочу, чтобы каждый игровой параметр (сила, ловкость, урон оружия, защита брони, скорость стрельбы) существовал в одном экземпляре в глобальном хранилище, чтобы изменения в одном месте мгновенно отражались во всех экранах.

#### Acceptance Criteria

1. WHEN параметр обновляется (например, `attributes.STR.value` увеличивается на 1 через перк) THEN итоговое значение `attributes.STR.total` SHALL пересчитываться автоматически без необходимости вызова `setAttributes` в каждом экране.
2. WHEN эффект истекает (например, `timedEffects` уменьшается) THEN зависимые параметры (здоровье, защита, бонусы) SHALL пересчитываться автоматически без ручного вызова `pruneExpiredTimedEffects` в каждом экране.
3. WHEN экран отображает параметр (например, здоровье или урон оружия) THEN он SHALL получать значение из глобального хранилища (например, `items['wep1'].damage.total`), а не вычислять его локально.
4. IF экран применяет эффект (например, перк или расходник) THEN эффект SHALL обновлять объект в хранилище, а не создавать локальный copy-paste данных.

---

### Требование 2: Нормализованная структура для всех типов параметров

**User Story:** Как разработчик, я хочу, чтобы все параметры хранились в нормализованном виде как словари ID → параметр, чтобы избежать дублирования и легко поддерживать связи между ними.

#### Acceptance Criteria

1. WHEN атрибут хранится THEN он SHALL быть в словаре `attributes[id]` с полями `base`, `modifiers[]`, `total`, а не в массиве `[{name: 'STR', value: 5}]`.
2. WHEN навык хранится THEN он SHALL быть в словаре `skills[id]` с полями `base`, `modifiers[]`, `total`, а не в массиве.
3. WHEN предмет хранится THEN он SHALL быть в словаре `items[id]` с вложенными параметрами (например, `damage: {base, modifiers[], total}`), а не просто объектом.
4. WHEN эффект хранится THEN он SHALL быть в словаре `effects[id]` с информацией о параметрах, которые он изменяет (например, `damageResistance: 5`, `maxHpBonus: 10`).

---

### Требование 3: Единый поток обновления всех параметров

**User Story:** Как пользователь, я хочу, чтобы изменение любого параметра (экипировка предмета, применение перка, истечение эффекта) обновляло итоговые значения автоматически, без необходимости вручную обновлять каждый экран.

#### Acceptance Criteria

1. WHEN пользователь экипирует предмет из инвентаря THEN `handleEquipWeapon` НЕ SHALL создавать новую копию предмета, а SHALL обновить `items[id].equipped = true`.
2. WHEN пользователь применяет перк, увеличивающий атрибут THEN `applyPerk` НЕ SHALL мутировать `attributes` напрямую, а SHALL добавить модификатор в `attributes[attrId].modifiers[]` и пересчитать `total`.
3. WHEN timed-эффект истекает THEN `pruneExpiredTimedEffects` НЕ SHALL просто удалять эффект, а SHALL обновить `effect.active = false` и триггерить пересчет зависимых параметров.
4. WHEN эффект применяется (например, расходник) THEN результат SHALL обновлять объект в `effects[id]`, а не возвращать локальный результат без сохранения.

---

### Требование 4: Интеграция с существующим `CharacterContext`

**User Story:** Как разработчик, я хочу, чтобы миграция на новую модель не ломала существующие функции и позволила постепенно обновлять компоненты.

#### Acceptance Criteria

1. WHEN `CharacterContext` загружает персонажа THEN он SHOULD создавать нормализованные структуры (`attributes`, `skills`, `items`, `effects`) из сохраненных данных, а не просто восстанавливать массивы.
2. IF `CharacterContext` сохраняет персонажа THEN он SHALL консолидировать нормализованные структуры обратно в формат базы данных (массивы `attributes`, `skills`, `equipment.items`, `equippedWeapons`).
3. WHEN старый компонент вызывает `setAttributes` или `setEquippedWeapons` THEN он SHOULD по-прежнему работать (с предупреждением в консоли о deprecated API).
4. WHEN новый компонент обновляет параметр THEN он SHALL использовать action-функции (например, `updateAttribute(attrId, delta)`, `equipItem(itemId)`), а не прямой вызов `setState`.

---

### Требование 5: Единая логика применения эффектов

**User Story:** Как разработчик, я хочу, чтобы все эффекты (черты, перки, timed-эффекты, моды) применялись через единую логику, а не дублировались в каждом экране.

#### Acceptance Criteria

1. WHEN эффект применяется (например, черта, перк, timed-эффект, мод оружия) THEN он SHALL добавлять модификатор в `modifiers[]` целевого параметра, а не мутировать `total` напрямую.
2. WHEN эффект отменяется (например, истекает или снимается) THEN он SHALL помечается как `active: false` или удаляется из `modifiers[]`, а не просто "пропускается" в логике экрана.
3. WHEN пересчитывается итоговое значение (например, `attributes.STR.total`) THEN система SHALL просматривать все активные модификаторы в `modifiers[]` и применять их (с учетом приоритетов).
4. WHEN эффект применяется к предмету (например, мод на оружие) THEN модификаторы (`damage`, `fireRate`, `qualities`) SHALL храниться внутри объекта предмета, а не в отдельном `modifiedItems` Map.

---

### Требование 6: Поддержка зависимостей между параметрами

**User Story:** Как пользователь, я хочу, чтобы изменения одного параметра (например, атрибута) автоматически обновляли зависимые значения (здоровье, защита, инициатива), без необходимости пересчитывать их вручную.

#### Acceptance Criteria

1. WHEN `attributes.STR.base` увеличивается THEN `carryWeight.total` SHALL пересчитываться автоматически (через `calculateCarryWeight(attributes, trait, ...)`).
2. WHEN `attributes.AGI.base` увеличивается THEN `initiative.total` SHALL пересчитываться автоматически (через `calculateInitiative(attributes)`).
3. WHEN `activeTimedEffects` меняется (эффект добавляется/истекает) THEN `maxHpBonus.total`, `damageResistance.total` SHALL пересчитываться автоматически.
4. WHEN `items[id].damage.total` меняется THEN экран `WeaponsAndArmorScreen` SHALL отображать новое значение без перезагрузки.

---

### Требование 7: Изоляция логики экранов

**User Story:** Как разработчик, я хочу, чтобы экраны управляли своими фильтрами и сортировкой локально, не влияя на глобальное состояние.

#### Acceptance Criteria

1. WHEN экран меняет фильтр (например, "показать только оружие с модами") THEN глобальное хранилище `items` НЕ SHALL изменяться.
2. WHEN экран применяет сортировку THEN он SHALL сортировать ID параметров локально (в `useMemo` или `useSelector`), а не менять порядок в хранилище.
3. WHEN состояние экрана меняется (фильтр, поиск) THEN Zustand Store НЕ SHOULD хранить фильтры всех экранов — каждый экран ИСПОЛЬЗУЕТ локальное состояние.
4. WHEN экран отображает параметры THEN он SHALL получать их из хранилища через селектор, фильтруя по своим критериям.

---

### Требование 8: Тестируемость и отладка

**User Story:** Как разработчик, я хочу, чтобы состояние всех параметров было легко тестировать и отлаживать.

#### Acceptance Criteria

1. WHEN состояние хранилища обновляется THEN изменения SHOULD быть видны в Redux DevTools (опционально для time-travel debugging).
2. WHEN тестируется логика применения эффекта THEN она SHALL быть отделена от React и тестируема как чистая функция (например, `applyModifier(base, modifiers)` в `domain/`).
3. WHEN происходит ошибка синхронизации THEN лог действия SHOULD содержать: ID параметра, старое и новое состояние, источник изменения (откуда вызван `updateParameter`).
4. WHEN эффект применяется THEN результат SHOULD быть проверяемым без рендеринга компонентов.

---

### Требование 9: Производительность

**User Story:** Как пользователь, я хочу, чтобы переходы между экранами были мгновенными, даже при большом количестве параметров.

#### Acceptance Criteria

1. WHEN экран отображает параметры THEN он SHALL получать только нужные данные через селектор (например, `useSelector(state => state.items.filter(i => i.equipped))`), а не весь `items`.
2. WHEN состояние обновляется THEN React SHOULD перерисовывать только компоненты, чьи данные изменились (благодаря shallow equality comparison в Zustand).
3. IF количество параметров (атрибуты, навыки, предметы, эффекты) превышает 1000 THEN прокрутка SHALL оставаться плавной (60 FPS).
4. WHEN эффект применяется THEN пересчет зависимых параметров SHOULD выполняться не более чем за O(n), где n — количество модификаторов для одного параметра.
